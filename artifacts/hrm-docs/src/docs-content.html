System Architecture

The system is a pnpm monorepo. All frontend artifacts are built statically and served by the central Express API server on port 8080. A single process handles both API requests and static file delivery.

monorepoworkspace/
├── artifacts/
│   ├── api-server/          # Express 5 API + static file serving (port 8080)
│   │   └── src/
│   │       ├── app.ts       # Route registration + static mounts
│   │       ├── routes/      # One file per resource
│   │       └── lib/auth.ts  # Token store + RBAC middleware
│   │
│   ├── hrm-skill-matrix/    # Main React/Vite app  →  /hrm-skill-matrix/
│   ├── hrm-dashboard/       # Analytics dashboard   →  /hrm-dashboard/
│   ├── hrm-pitch-deck/      # Slides deck           →  /hrm-pitch-deck/
│   └── hrm-docs/            # This docs site        →  /hrm-docs/
│
└── lib/
    ├── api-spec/            # OpenAPI 3.1 spec (source of truth)
    ├── api-client-react/    # Generated TanStack Query hooks
    ├── api-zod/             # Generated Zod validation schemas
    └── db/                  # Drizzle ORM schema + Postgres client

Port routing: Only ports 8080 and 8081 are exposed. All artifacts are built statically (pnpm run build) with their BASE_PATH baked in, then served from the API server via Express static middleware.

Database — Row Level Security (RLS)

When deployed to Supabase, all 9 tables have RLS enabled. Each table has policies that enforce the same access rules as the Express middleware layer, providing defence-in-depth at the database level.

Note: The Replit deployment uses Express middleware for auth (token-based). The RLS policies below apply to Supabase/direct PostgreSQL access and use auth.uid() from Supabase Auth. Both layers enforce the same business rules.

sql-- Enable RLS on all tables
ALTER TABLE public.users              ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.departments         ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.employees           ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.skills              ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.evaluation_campaigns ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.evaluations          ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.evaluation_summaries ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.training_recommendations ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.audit_logs           ENABLE ROW LEVEL SECURITY;

-- POLICY 1: super_admin — full access to every table
CREATE POLICY super_admin_all ON public.users
  FOR ALL USING (
    (SELECT role FROM public.users WHERE id = auth.uid()) = 'super_admin'
  );

-- Same pattern on departments, employees, skills, campaigns, evaluations…

-- POLICY 2: dept_head — own department employees only
CREATE POLICY dept_head_view_employees ON public.employees
  FOR SELECT USING (
    department_id = (SELECT department_id FROM public.users WHERE id = auth.uid())
    OR (SELECT role FROM public.users WHERE id = auth.uid()) = 'dept_head'
  );

-- dept_head can only write evaluations for their dept in Active campaigns
CREATE POLICY dept_head_edit_evaluations ON public.evaluations
  FOR ALL USING (
    (SELECT role FROM public.users WHERE id = auth.uid()) = 'dept_head'
    AND employee_id IN (
      SELECT id FROM public.employees
      WHERE department_id = (SELECT department_id FROM public.users WHERE id = auth.uid())
    )
    AND campaign_id IN (
      SELECT id FROM public.evaluation_campaigns WHERE status = 'Active'
    )
  );

-- POLICY 3: hr_coordinator — read-only access to all data
CREATE POLICY hr_readonly_all ON public.employees
  FOR SELECT USING (
    (SELECT role FROM public.users WHERE id = auth.uid())
      IN ('hr_coordinator', 'super_admin')
  );

-- POLICY 4: employee — own profile only
CREATE POLICY employee_view_own ON public.employees
  FOR SELECT USING (
    user_id = auth.uid()
    OR (SELECT role FROM public.users WHERE id = auth.uid())
      IN ('hr_coordinator', 'dept_head', 'super_admin')
  );

Typical Workflow

1
Set up departments & skills
Navigate to Departments → add each of the 9 departments. Then Skills → add the skill library for each department with appropriate weights and criticality levels.
2
Import employees
Go to Employees → Create for each employee, or use the bulk import feature to import from Excel. Assign each employee to their department.
3
Create user accounts
Go to Users → Create accounts for each dept head and HR coordinator. Assign their role and department. They will use email/password to log in.
4
Launch an evaluation campaign
Campaigns → Create → set title, type, date range → Save as Draft → Activate when ready. Notify department heads to begin scoring.
5
Complete and review
Once all scores are in, set campaign status to Completed. The system auto-calculates summaries and generates training recommendations. Review the dashboard for results.

User Guide — Department Head

🏭
Department Head
Evaluates employees in own department only. Scoped access.
dept_head
  • View employees in own department
  • Submit and edit skill scores during active campaigns
  • View own department's performance metrics
  • View and update training recommendations for own employees
  • Assign trainers for recommendations in their department
  • Cannot see other departments' data
  • Cannot create campaigns or manage users
  • Cannot access audit logs

Evaluating Employees

  1. Log in with department head credentials.
  2. Go to Campaigns — you will see only campaigns assigned to your department (or company-wide ones).
  3. Click an Active campaign → open the Skill Matrix Grid.
  4. Each row is an employee; each column is a skill. Enter scores from 0–4 for each cell.
  5. Scores are saved immediately. The employee summary recalculates in real-time.
  6. When all employees are scored, inform HR/Admin to close the campaign.

Scoring & Calculations

The hrm scoring system uses a weighted average of skill scores on a 0–4 scale, converting the result to a percentage for class assignment.

Score Level Descriptors

ScoreLevelDescription
0Cannot performEmployee has no demonstrated ability in this skill
1Constant supervisionCan attempt the task but requires continuous guidance
2Occasional supervisionCan complete the task with periodic check-ins
3IndependentWorks competently without supervision
4Expert / TrainerMastery level; can train others in this skill

Weighted Score Formula

formulapercentage = Σ(score × weight) / Σ(4 × weight) × 100

Where:
  score  = individual skill score (0–4)
  weight = skill importance weight (1–5)
  4      = maximum possible score per skill

Performance Classes

A
Score ≥ 85%
High performer. Fully competent. Eligible for leadership development and promotion track.
B
Score 60–84%
Developing. Core skills present. Targeted training recommended to close gaps.
C
Score < 60%
Needs improvement. Immediate training required. Priority for intervention.

Implementation (TypeScript)

typescriptfunction calculateSummary(
  skillScores: Array<{ skill_id: string; score: number; weight: number }>
) {
  let totalScore = 0;
  let maxScore = 0;

  for (const { score, weight } of skillScores) {
    totalScore += score * weight;
    maxScore  += 4 * weight;
  }

  const percentage = maxScore > 0
    ? (totalScore / maxScore) * 100
    : 0;

  const evalClass: "A" | "B" | "C" =
    percentage >= 85 ? "A" :
    percentage >= 60 ? "B" : "C";

  return { total_score: totalScore, max_possible_score: maxScore,
            percentage: Math.round(percentage * 10) / 10, class: evalClass };
}

Completed → Archivedsuper_admin, hrCampaign moves to historical view only; no further changes

Note: Campaigns in Draft or Archived status reject evaluation submissions with HTTP 400. Only Active campaigns accept scores.

Campaign Types

  • Monthly — Regular monthly assessments, usually department-specific.
  • Quarterly — Quarterly company-wide evaluation.
  • Bi-Annually — Mid-year and end-of-year comprehensive review.
  • Custom — Ad-hoc campaigns for special projects or new hire assessments.

Training Recommendations

Training recommendations are auto-generated when evaluation scores are submitted. The recommendation type is determined by the skill score and the overall employee class.

Auto-Generation Rules

Skill score = 2 (Occasional supervision)Short-termMedium — development plan

When an employee's evaluation is re-submitted (scores updated), all existing Pending recommendations for that employee+campaign are deleted and regenerated from scratch. This prevents duplicate training plans.

Data Migration

For organisations migrating from the Excel-based system, follow this sequence to minimise downtime.

1
Export clean data from Excel
Export Departments, Employees, and Skills sheets as CSV. Verify there are no blank rows, duplicate employee codes, or missing department references.
2
Create departments via API
POST each department. Note the returned UUID — you will need it for employee records.
3
Import skills per department
POST each skill with the correct department_id, weight, and criticality. Use code field to match back to Excel columns (e.g. UPH-01).
4
Import employees
POST each employee. Map Excel "Department Name" to the correct UUID from step 2. Preserve employee_code values for traceability.
5
Create historical evaluation campaign
Create a campaign titled "Historical Import" with type Custom, status Draft. Set the date range to cover your oldest Excel data.
6
Bulk submit historical scores
Use POST /api/evaluations/bulk to submit all historical scores. The system will calculate summaries and set initial performance classes.
7
Complete the historical campaign
Set the campaign to Completed. This finalises all current_class values on employee records. Verify counts match your Excel totals.

Tip: Use a Node.js migration script for bulk imports to avoid hitting rate limits. The API accepts JSON payloads; a simple for...of loop with await fetch() works well.

Production Checklist

  • Environment variables — ensure DATABASE_URL and PORT=8080 are set in production.
  • Static builds — all four artifacts must be built before starting the API server.
  • Token persistence — the current in-memory token store does not survive restarts. For production, replace with Redis or a database-backed session store.
  • Password hashing — the demo uses plaintext passwords. Production deployments must hash passwords with bcrypt or Argon2.
  • CORS — configure cors() origin allowlist to restrict to your domain.
  • HTTPS — deploy behind a reverse proxy (Nginx, Cloudflare) with TLS termination.
  • Audit logs — verify audit_logs table is present and is not purged by maintenance jobs.
